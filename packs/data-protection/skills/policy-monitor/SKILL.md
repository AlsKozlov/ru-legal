# /policy-monitor (drift detection политики vs practice)

## Назначение

Политика обработки ПДн **обязательна** для оператора по ст.18.1 152-ФЗ — оператор
должен опубликовать политику в свободном доступе. Но политика быстро **drift'ит**
от того, что компания фактически делает.

Этот skill находит drift'ы — где политика **не соответствует** practice — и
предлагает обновления.

**Два режима:**
- **Sweep mode** (default, `--sweep`): periodic crawl saved outputs (ОВПД, ДОУ
  reviews, use-case-triage results) и сравнение с политикой
- **Direct query**: проверка конкретной predposed activity vs политика

---

## Sweep mode

### Шаг 1. Загрузи политику

Прочитай PROFILE.md → `## Политика обработки ПДн → URL` или текст. Если ссылка —
fetch текст (через WebFetch tool).

### Шаг 2. Crawl outputs folder

Read directory specified в PROFILE.md → `## Стиль работы → Output path`. Look for:

- `OVPD-*.md` — оценки вреда
- `dpa-review-*.md` — review договоров обработки
- `use-case-triage-*.md` — privacy triage
- `dsar-*.md` — DSAR responses

Limit period: last 90 days (или per user request).

### Шаг 3. Extract data points

Из каждого output extract:

- Какие категории ПДн обрабатываются?
- Какие цели обработки?
- Какие vendor'ы / sub-processors?
- Cross-border ли?
- Retention периоды?
- Согласия / правовые основания?

### Шаг 4. Compare к политике

Для каждой data point — check:

- **Покрыта ли политикой?** Найди соответствующий пункт.
- **Если найдено — совпадает ли?** Например, политика говорит "хранение 3 года"
  vs ОВПД говорит "5 лет".
- **Если не найдено — gap:** activity не покрыта политикой.

### Шаг 5. Classify gaps

| Тип gap'а | Пример | Severity |
|-----------|--------|----------|
| **Категория ПДн отсутствует** | Политика не упоминает биометрию, но компания внедрила биометрический турникет | 🔴 (нарушение ст.18.1) |
| **Цель отсутствует** | Политика не упоминает маркетинг, но компания шлёт рассылки | 🔴 |
| **Vendor / sub-processor отсутствует** | Политика не указывает Yandex Cloud, но компания там хостится | 🟡 |
| **Cross-border не упомянут** | Политика не описывает передачу за рубеж, но фактически передаётся | 🔴 |
| **Retention несовпадение** | Политика говорит 3 года, ДОУ — 5 лет | 🟡 |
| **Сроки реализации прав не указаны** | Политика silent про ст.21 сроки | 🟢 рекомендация |

### Шаг 6. Output: drift report

```markdown
# Privacy Policy Drift Report — [DD.MM.YYYY]

## Summary
- Activities reviewed: [N] (за последние 90 дней)
- Gaps found: [N]
  - 🔴 Critical (нарушение 152-ФЗ): [N]
  - 🟡 Internal inconsistency: [N]
  - 🟢 Improvement opportunity: [N]

## Critical gaps

### Gap 1: [Category]

**Activity (источник):** [например — ОВПД от 2026-04-15 про биометрический турникет]
**Что обрабатывается:** Биометрические ПДн (отпечатки)
**Политика говорит:** "[цитата с пометкой 'silent'/'противоречит']"
**Risk:** Нарушение ст.18.1 152-ФЗ + ст.11 (биометрия требует упоминания)

**Recommended action:**
- [ ] Обновить политику в секции 2.1 "Категории обрабатываемых ПДн" — добавить
      "биометрические данные сотрудников для целей контроля доступа"
- [ ] Опубликовать на сайте до [DD.MM.YYYY]
- [ ] Уведомить сотрудников об обновлении

## Internal inconsistencies (🟡)

[...]

## Improvement opportunities (🟢)

[...]

## Next steps

1. **DPO review** этого отчёта в течение 7 дней
2. **Update политики** для всех 🔴 gaps в течение 14 дней
3. **Publish** updated политику + уведомить субъектов о существенных изменениях
4. **Re-run** этот skill через 90 дней
```

## Direct query mode

### Использование

```
/data-protection:policy-monitor "Мы хотим начать использовать AI для матчинга
кандидатов — какие изменения политики нужны?"
```

### Workflow

1. Загрузи политику обработки ПДн
2. Понять предложенную practice (clarify если vague)
3. Find related sections в политике
4. Check coverage

Output:

```markdown
## Policy check: [Описание new practice]

**Покрыто ли политикой?** Да / Нет / Частично

[Если "Да"]
**Применимые пункты:** [цитаты]
**Достаточно ли:** [Yes / No — что добавить]

[Если "Нет" или "Частично"]
**Что отсутствует:**
- [Конкретный gap]

**Recommended обновления политики:**
- [ ] Добавить в раздел [X] следующий текст: "[предложенный текст]"
- [ ] [Другие изменения]

**Доп.compliance шаги:**
- [ ] ОВПД (если триггер): запустить `/data-protection:pia-generation`
- [ ] Обновить согласия субъектов (если требуется)
- [ ] Уведомить РКН (если изменения существенные)
- [ ] ДОУ с vendor'ами (если применимо)
```

## Что важно для РФ контекста

### Публикация политики обязательна

Ст.18.1 152-ФЗ требует, чтобы оператор **опубликовал** политику в свободном доступе:
- На сайте (если есть)
- Иным способом обеспечить доступ субъектам

Политика **не на сайте** = нарушение, штраф 30-60 тыс руб для юрлица.

### Обязательное содержание политики (по приказу РКН)

- Цели обработки
- Категории ПДн
- Категории субъектов
- Способы обработки
- Сроки обработки и хранения
- Правовые основания
- Информация о трансграничной передаче
- Сведения об обеспечении безопасности
- Права субъектов и порядок их реализации
- Контакты оператора (DPO)

### Когда нужно уведомить субъектов о изменениях

Существенные изменения политики → уведомление субъектов рекомендуется (хотя 152-ФЗ
прямо не требует). Способы:
- Email рассылка (для тех у кого есть email)
- Уведомление в личном кабинете
- Объявление на сайте

При изменении **правовых оснований** (например, добавление новой цели) — может
потребоваться новое согласие субъекта.

## Что этот skill НЕ делает

- Не обновляет политику автоматически — только flag'ает gaps
- Не публикует обновления — это работа DPO + IT (на сайт)
- Не валидирует юридическую корректность — DPO + GC review
- Не делает legal opinion при сложных кейсах
