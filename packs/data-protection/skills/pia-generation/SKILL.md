# /pia-generation (ОВПД по ст.18.1 152-ФЗ)

## Назначение

**ОВПД** (Оценка вреда персональным данным) — российский аналог GDPR DPIA по
ст.18.1 152-ФЗ. Обязательна с 01.09.2022 (ФЗ-266) для определённых типов обработки.

## Что такое ОВПД (РФ specifics)

Ст.18.1 152-ФЗ обязывает оператора:

1. Определять угрозы безопасности ПДн при обработке
2. Применять правовые, организационные и технические меры защиты
3. **Оценивать вред** субъектам ПДн в случае нарушения требований

**В отличие от GDPR DPIA** — ОВПД более operational document для **внутреннего**
комплаенса, не для согласования с регулятором. РКН проверяет наличие ОВПД при
проверках.

## Когда нужна ОВПД (обязательные случаи)

См. `/data-protection:use-case-triage` — оно определяет нужна ли ОВПД.

Обязательные триггеры:
1. Спец.категории ПДн (ст.10)
2. Биометрические ПДн (ст.11)
3. ПДн несовершеннолетних
4. Автоматизированное решение (ст.16)
5. Cross-border передача (ст.12)
6. Поручение обработки sub-processor'у (ст.6 ч.3)

## Workflow

### Шаг 1. Загрузи house style

PROFILE.md → `## ОВПД house style`:
- Triggers (что заставляет делать ОВПД)
- Structure (sections)
- Depth (level of detail)
- Sign-off (кем)

### Шаг 2. Intake questions

Если пользователь дал только название фичи — спроси:

- **Описание деятельности:** что конкретно будем делать с ПДн?
- **Цель обработки:** какую бизнес-проблему решаем?
- **Категории ПДн:** конкретные данные (ФИО, email, биометрия, и т.д.)
- **Категории субъектов:** клиенты / сотрудники / другие
- **Масштаб:** количество субъектов (приближённо)
- **Системы:** где будут храниться данные? Локализация?
- **Vendor'ы:** кто ещё имеет доступ?
- **Согласия:** есть ли уже / будут собираться?
- **Сроки хранения:** сколько лет / в зависимости от чего?
- **Автоматизированное решение:** да/нет; если да — какое?
- **Cross-border:** будет ли передача за рубеж?

### Шаг 3. Анализ угроз и оценка вреда

Для каждой категории ПДн — оцени:

| Угроза | Вероятность | Тяжесть | Оценка вреда |
|--------|-------------|---------|--------------|
| Утечка ПДн (incident) | Низкая/Средняя/Высокая | Низкий/Средний/Тяжкий | [например, моральный вред + штрафы РКН] |
| Несанкционированный доступ внутри организации | | | |
| Передача в недружественные страны | | | |
| Использование вне целей | | | |
| Долгое хранение сверх необходимости | | | |
| Утрата данных (resilience) | | | |
| Манипуляция данными (целостность) | | | |

### Шаг 4. Меры защиты (ст.19 152-ФЗ)

Уровень защиты ПДн определяется по Постановлению Правительства РФ от 01.11.2012
№1119:

| Уровень | Когда применяется |
|---------|-------------------|
| **УЗ-1** (наивысший) | Спец.категории > 100 тыс субъектов, или биометрия > 100 тыс |
| **УЗ-2** | Спец.категории ≤ 100 тыс, или биометрия ≤ 100 тыс, или иные > 100 тыс |
| **УЗ-3** | Иные категории ПДн в больших объёмах |
| **УЗ-4** (минимальный) | Иные категории в малых объёмах, обработка только сотрудников оператора |

Для каждого уровня — конкретные технические меры по приказам ФСТЭК / ФСБ.

В ОВПД укажи:
- **Определённый уровень защиты**
- **Конкретные меры** (организационные + технические)
- **Сертифицированные средства защиты** (если УЗ-1 / УЗ-2)

ОВПД должен быть утверждён:

- **Гендиректор** — для high-risk activities

## Что этот skill НЕ делает

- Не одобряет ОВПД сам — только draft'ит, sign-off за DPO
- Не валидирует actual меры безопасности — это compliance audit
- Не подает уведомления РКН — отдельный процесс
- Не делает legal opinion — это работа юр.отдела при complex cases
