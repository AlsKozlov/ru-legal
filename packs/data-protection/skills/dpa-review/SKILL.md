# /dpa-review (ДОУ review по 152-ФЗ)

## Назначение

ДОУ (Договор об обработке персональных данных, в России также именуемый "поручение
на обработку ПДн" по ст.6 ч.3 152-ФЗ) приходят в двух flavor'ах, и review почти
противоположен для каждого:

- **Customer прислал свой ДОУ нам** → мы — лицо, осуществляющее обработку по
  поручению → защищаем operational flexibility
- **Vendor прислал ДОУ нам / мы шлём ему** → мы — оператор → защищаем свои (и
  клиентов) данные

Оба review читают из **одного** playbook PROFILE.md но из **противоположных** строк.

---

## Critical РФ context

### 1. Юридическая основа — ст.6 ч.3 152-ФЗ

**Ст.6 ч.3:** "Оператор вправе поручить обработку персональных данных другому лицу
с согласия субъекта персональных данных, если иное не предусмотрено федеральным
законом, на основании заключаемого с этим лицом договора, в том числе государственного
или муниципального контракта, либо путём принятия соответствующего акта государственным
или муниципальным органом."

**Обязательные условия в ДОУ (ст.6 ч.3):**

1. **Перечень действий** с ПДн, которые будут совершаться лицом по поручению
2. **Цели обработки**
3. **Обязанность** лица, осуществляющего обработку, соблюдать **конфиденциальность**
   ПДн
4. **Обязанность** обеспечить безопасность ПДн при их обработке
5. **Требования к защите** в соответствии со ст.19 152-ФЗ

**🔴 КРИТИЧНО:** ДОУ без любого из этих 5 элементов — **формально не соответствует
152-ФЗ**. РКН при проверке квалифицирует как нарушение.

### 2. Терминология (отличается от US/EU)

| US/EU | РФ |
|-------|-----|
| Controller | **Оператор** (ст.3 п.2) — определяет цели обработки |
| Processor | **Лицо, осуществляющее обработку по поручению** (ст.6 ч.3) |
| Sub-processor | Sub-processor поручения — допустимо при определённых условиях |
| DPA / Data Processing Agreement | **ДОУ** (Договор об обработке) или **Поручение** |
| SCC (Standard Contractual Clauses) | Нет прямого аналога — для cross-border используется ст.12 152-ФЗ |
| Joint Controllers | Концепция существует, но менее развита |

### 3. Ответственность

**Оператор отвечает за всю цепочку обработки** (ст.6 ч.3 + ст.17). Даже если
обработка произведена sub-processor'ом — оператор несёт перед субъектом.

Это **полная ответственность** — sub-processor не освобождает оператора.

### 4. Локализация (ст.18 ч.5) — обязательная в ДОУ

Если ДОУ покрывает ПДн **российских граждан** — первичный сбор + хранение
**обязательно в РФ**. Это критичный element ДОУ.

Штраф с 30.05.2025: до 10 млн руб (повторно — до 18 млн).

## Workflow

### Шаг 1. Определи направление (critical first step)

Спроси если неясно:

- **Мы — лицо, осуществляющее обработку?** (customer/контрагент прислал нам своё
  ДОУ для подписи) → читай "Когда мы обработчик" таблицу
- **Мы — оператор?** (мы шлём ДОУ vendor'у, или review его ДОУ для подписи нами) →
  читай "Когда мы оператор" таблицу

**Если неясно — ASK.** Перевернёшь направление = invertешь все рекомендации.

### Шаг 2. Pre-flight

1. Прочитай PROFILE.md → playbook ДОУ (sections для оператора и для обработчика)
2. Загрузи политику обработки ПДн → ДОУ не может противоречить
3. Load prior outputs на этого контрагента / activity:
   - Prior `use-case-triage` results
   - Prior `pia-generation` outputs
   - Prior `dpa-review` outputs

Если prior output найден — cite в review (severity floor: triage 🔴 не может быть
quietly downgraded к 🟢 без объяснения).

### Шаг 3. Check секторальные overlays

**Перед term-by-term review** — спроси:

> **Sectoral overlays для РФ — проверь сразу:**
>
> Включает ли обработка по этому ДОУ:
>
> - **Финансовые данные клиентов** (банковская тайна по 395-1 ст.26)?
>   ДОУ нужен specific режим. Банковская тайна имеет substantive ограничения сверх
>   152-ФЗ.
> - **Медицинские данные** (323-ФЗ ст.13)? ДОУ должен учитывать врачебную тайну —
>   уведомление субъекта о disclosure обязательно.
> - **Данные о услугах связи** (тайна связи 126-ФЗ ст.63)?
> - **Особые категории ПДн** (ст.10 152-ФЗ — здоровье, политика, биометрия,
>   генетика)? Усиленные требования к ИБ + согласия.
> - **Биометрические ПДн** (ст.11 + 572-ФЗ ЕБС)? Особый режим, ст.19 уровень
>   защиты повышенный.
> - **ПДн несовершеннолетних** (ст.9 ч.4)?

Если **да** — sectoral norm supplies controlling restrictions. Research конкретные
provisions и cite в review.

## Term-by-term review

### Core terms — check каждый ДОУ

#### 1. **Identification of parties и направление**

🟢 OK if:
- Чётко указаны оператор vs лицо, осуществляющее обработку
- Указано что одна сторона "поручает", другая "принимает" обработку

🟡 YELLOW if:
- Direction implicit but inconsistent
- "Стороны взаимно обрабатывают" — неясно роли

🔴 RED if:
- Direction не указано
- ДОУ имеет статус "взаимный" — но 152-ФЗ ст.6 ч.3 требует чёткой иерархии

#### 2. **Перечень действий с ПДн** (ст.6 ч.3 п.2)

Какие конкретные действия совершает обработчик:

- сбор
- запись
- систематизация
- накопление
- хранение
- уточнение (обновление, изменение)
- извлечение
- использование
- передача (распространение, предоставление, доступ)
- обезличивание
- блокирование
- удаление
- уничтожение

🟢 GREEN: explicit список конкретных действий

🟡 YELLOW: "иные действия" / "включая, но не ограничиваясь" без конкретики

🔴 RED: "all data processing" / vague description

#### 3. **Цели обработки** (ст.6 ч.3 + ст.5 ч.2)

Каждая цель должна быть **конкретной, заранее определённой**.

🟢 GREEN: цели explicit, привязаны к функциональности (например "обработка платежей
по договору № X", "доставка email рассылок")

🟡 YELLOW: цели слишком broad ("оказание услуг") без конкретики

🔴 RED: цели не указаны или maximally vague

#### 4. **Конфиденциальность** (ст.6 ч.3 п.4)

🟢 GREEN: explicit обязательство обработчика **не раскрывать** ПДн третьим лицам
без согласия оператора + обязательство **не использовать** для своих целей

🟡 YELLOW: обязательство есть, но без указания мер ("обеспечить конфиденциальность"
без специфики)

🔴 RED: конфиденциальность не упомянута / обработчик имеет право использовать ПДн в
своих интересах

#### 5. **Меры безопасности** (ст.6 ч.3 п.5 + ст.19)

Ст.19 152-ФЗ + Постановление Правительства РФ от 01.11.2012 №1119 (об уровнях
защиты ПДн).

🟢 GREEN:
- Указан уровень защиты (УЗ-1 / УЗ-2 / УЗ-3 / УЗ-4) или обязательство соблюдать
  определённый уровень
- Соответствие требованиям ФСТЭК / ФСБ упомянуто
- Обязательство применять сертифицированные средства защиты (если для УЗ-1/2)

🟡 YELLOW: общие фразы "обеспечить безопасность" без конкретики; не указан уровень

🔴 RED:
- Меры безопасности не описаны
- Уровень защиты несоответствует категории ПДн (например, обрабатывают спец.категории
  на УЗ-4)

#### 6. **Локализация** (ст.18 ч.5) — критично

🟢 GREEN: явное обязательство хранить и обрабатывать ПДн **первично в РФ**

🟡 YELLOW: упомянуто географически (например "хранение на серверах в Москве") но
без указания на 152-ФЗ ст.18 ч.5

🔴 RED:
- Локализация в РФ не указана
- Обработчик использует cloud за рубежом для primary storage ПДн российских граждан

**Если RED:** **deal-breaker**. Штрафы до 18 млн руб + риск приостановки операции.

#### 7. **Cross-border передача** (ст.12)

🟢 GREEN: либо не происходит, либо включены conditions:
- Указаны страны передачи
- Согласие субъектов на cross-border получено
- Уведомление РКН поданы (двухэтапное с 03.2023)

🟡 YELLOW: cross-border происходит, но procedure не описан

🔴 RED: cross-border в недружественные страны без согласий / уведомлений

#### 8. **Subprocessors / поручение sub-обработчику**

Обработчик может привлекать sub-processor'ов **только** с согласия оператора (или
если ДОУ прямо разрешает).

🟢 GREEN:
- Список sub-processor'ов explicit или приложение
- Механизм одобрения новых sub-processor'ов (уведомление за N дней + право на
  возражение)
- Каскадные обязательства sub-processor'а по 152-ФЗ explicit

🟡 YELLOW:
- Sub-processor'ы упомянуты но без списка
- Уведомление о смене sub-processor'а < 30 дней (РФ стандарт)

🔴 RED:
- Обработчик имеет право свободно привлекать любых sub-processor'ов
- Sub-processor'ы не несут обязательств по 152-ФЗ

#### 9. **Уведомление об инцидентах** (ст.21 ч.4 152-ФЗ — для оператора + новая
практика РКН по утечкам)

🟢 GREEN:
- Срок уведомления оператора об инциденте — **24 часа** (новая практика РКН с 2025)
- Содержание уведомления explicit (категории субъектов, объём ПДн, меры
  реагирования)
- Каскадная ответственность за невуведомление

🟡 YELLOW: срок 48-72 часа (US/EU стандарт) — не нарушение, но slower чем РКН
ожидает

🔴 RED:
- Срок > 72 часа
- Уведомление "по мере возможности"
- Нет требования информировать оператора

#### 10. **Аудит и проверки**

🟢 GREEN: оператор имеет право проверить обработчика — на месте или через
самосертификацию

🟡 YELLOW: только self-attestation (обработчик сам отчитывается); не право на
выездную проверку

🔴 RED: оператор не имеет права проверять обработчика

#### 11. **Удаление / возврат данных при прекращении договора**

🟢 GREEN:
- Срок удаления explicit (рекомендация — 30-90 дней)
- Возможность возврата данных оператору (для миграции)
- Подтверждение удаления документально

🟡 YELLOW:
- Срок > 90 дней
- "В разумный срок" без конкретики

🔴 RED:
- Обработчик имеет право сохранить данные после прекращения
- Срок > 1 года без основания

#### 12. **Срок действия и расторжение**

🟢 GREEN: ДОУ действует до окончания основного договора + срок удаления
данных

🟡 YELLOW: автоматическое продление без notice (не нарушение, но риск)

🔴 RED: бессрочный ДОУ без права расторжения

## Privacy policy consistency check

После term-by-term review — check ДОУ vs политика обработки ПДн (PROFILE.md →
`## Политика обработки ПДн`).

**Common conflicts:**

- Политика говорит "ПДн в РФ" — ДОУ позволяет обработку за рубежом
- Политика говорит "только для целей договора" — ДОУ permit'ит broader purposes
- Политика говорит "уведомление о sub-processor'ах за 60 дней" — ДОУ — 14 дней
- Политика "не передаём в недружественные страны" — ДОУ permit'ит cross-border

Если **direct conflict** → обновить политику **до** подписания ДОУ, или
изменить ДОУ.

## Specific check tables — по направлению

### Когда мы — оператор (наши данные в их руки)

**Дополнительно проверь:**

- Обработчик имеет certification ФСТЭК / соответствие УЗ требуемому?
- Обработчик имеет infrastructure в РФ (для первичного хранения)?
- В случае инцидента — кто несёт ответственность перед нашими субъектами? (Мы, но
  каскадные требования к обработчику есть?)
- Право на выездную проверку обработчика?
- Удаление данных при terminate в течение 30-60 дней?

### Когда мы — обработчик (data клиента в нашей системе)

**Дополнительно проверь:**

- Defensible scope действий — не позволяем клиенту требовать что-то невозможное?
- Уведомления об инцидентах — реалистичные сроки (24-48 ч feasible)?
- Audit rights клиента — limited to reasonable timeframes / scope?
- Subprocessor rights — наш right использовать стандартный stack (Yandex Cloud / VK
  Cloud)?
- Limited liability cap explicit?

## Что этот skill НЕ делает

- Не подписывает ДОУ — final approval за DPO + GC
- Не проверяет реальное соблюдение мер безопасности обработчиком — это compliance
  audit отдельно
- Не валидирует сертификаты ФСТЭК / ФСБ — manual verification
- Не валидирует actual локализацию infrastructure — это IT compliance

## Реальные кейсы РФ practice

### Pattern 1: ДОУ отсутствует — нарушение даже без incident'а

РКН квалифицирует **отсутствие** ДОУ при поручении обработки как нарушение ст.6
ч.3. Штраф 60-100 тыс (юрлицо). При наличии утечки — multiplied.

### Pattern 2: Локализация — биrolаций кейс

Если ДОУ не содержит локализации в РФ и фактически данные за рубежом — штраф до 10
млн руб (повторно — до 18 млн). Российские суды строго применяют ст.18 ч.5.

### Pattern 3: Sub-processor'ы без согласия оператора

РКН квалифицирует use sub-processor'ов без согласия оператора как нарушение ст.6
ч.3. Каскадные штрафы — на оператора + на обработчика.

## Disclaimer

Этот skill производит preliminary анализ для DPO + GC. Не legal advice.

**Особо важно:**
- ДОУ без 5 обязательных элементов ст.6 ч.3 — auto-RED, no exceptions

  specialist обязателен
