# /contract-review

Полная юридическая review коммерческого договора по российскому праву.

---

## Когда запускать

- Пользователь явно даёт путь к файлу договора (`.docx`, `.pdf`, `.txt`, `.md`).
- Пользователь просит "проверить договор", "сделать review", "проанализировать соглашение".
- Из `/cold-start-interview` после успешной настройки профиля (по запросу пользователя).

Если файл не указан и в контексте нет договора — попроси пользователя дать путь.

---

## Pre-flight checks

Прежде чем начать — обязательно проверь:

1. **Profile существует** → читаем `~/.claude/plugins/config/contract-law/CLAUDE.md`.
   Если profile отсутствует или содержит placeholder'ы `[заполнить]` → остановись и попроси: "Запусти сначала `/contract-law:cold-start-interview` чтобы я знал ваш контекст. Без этого review будет generic."

2. **MCP `pravo` доступен?** Пробуем `pravo.search_npa(query="ГК РФ", limit=1)`. Если 404/timeout → продолжаем в degraded mode, но **во всех цитатах ставим флаг `[ред. требует проверки]`** и в финальном отчёте предупреждаем.

3. **Файл договора существует и читается?** Если PDF — нужно убедиться, что текст экстрактнулся, не сканированная картинка без OCR.

## Workflow

### Шаг 1. Подготовка контекста

1.1. Прочитай practice profile из `~/.claude/plugins/config/contract-law/CLAUDE.md`. Извлеки:
- Sell-side / buy-side позицию
- Целевой LoL
- Indemnity готовность
- The one thing (главное no-go условие)
- Эскалационная матрица (для определения, на каком уровне решать YELLOW/RED)

1.2. Прочитай договор полностью. Извлеки структурно:
- Стороны (полное наименование + ИНН/ОГРН если указаны)
- Тип договора (если не задан через `--contract-type`)
- Предмет
- Сумма / цена
- Срок действия + auto-renewal
- Применимое право и подсудность
- Все номера пунктов в иерархии (1, 1.1, 1.1.1)

1.3. Если ИНН/ОГРН контрагента есть **и доступен `egrul-mcp`** — вызови `egrul.lookup_company(inn)`. Извлеки:
- Статус (действующее / в стадии ликвидации / банкрот / прекращено)
- Дата регистрации (срок существования)
- Руководитель (ФИО, ИНН)
- Бенефициары если есть
- Изменения за последние 12 месяцев

1.4. Если ИНН/ОГРН есть **и доступен `kad-mcp`** — вызови `kad.search_cases(party_inn=inn, limit=20, date_from=YYYY-MM-DD)` за последние 3 года. Извлеки:
- Общее число дел
- Соотношение истец/ответчик
- Топ-3 типа споров
- Признаки банкротства (дела по 127-ФЗ)

### Шаг 2. Параграф-by-параграф review

Для **каждого** пункта договора (1.1, 1.2, ..., N.N):

#### 2.1. Identify clause type

Определи тип пункта:
- `predmet` — предмет договора
- `cena` — цена и порядок расчётов
- `sroki` — сроки исполнения
- `obyazannosti` — обязанности сторон
- `otvetstvennost` — ответственность (LoL, штрафы, неустойка)
- `garantii` — гарантии и заверения
- `force-majeure` — обстоятельства непреодолимой силы
- `konfidentsialnost` — конфиденциальность
- `intellectual-property` — IP
- `personal-data` — обработка ПДн (152-ФЗ)
- `razreshenie-sporov` — разрешение споров (подсудность, применимое право)
- `srok-deistviya` — срок действия и порядок прекращения
- `prochee` — заключительные положения

#### 2.2. Check против ГК РФ

Через `pravo.search_npa()` найди релевантные статьи ГК РФ. Для договорных типов:

| Тип договора | Главы ГК РФ |
|--------------|-------------|
| Купля-продажа | гл. 30 |
| Поставка | §3 гл. 30 (ст. 506-524) |
| Услуги | гл. 39 |
| Подряд | гл. 37 |
| Аренда | гл. 34 |
| Лицензионный | ст. 1233-1239, ч.4 ГК |
| Заём | гл. 42 |
| Хранение | гл. 47 |
| NDA / коммерческая тайна | 98-ФЗ + ст. 1465 ГК ч.4 |

Общие положения о договорах: гл. 27-29 (ст. 420-453).

Проверь:
- **Императивные нормы** — пункт не должен им противоречить. Если противоречит → 🔴.
- **Диспозитивные нормы** — пункт может отступать, но проверь обоснованность.
- **Существенные условия** — для каждого типа договора ГК указывает существенные условия (например, для поставки — наименование товара, количество, срок; ст. 506, 455 ГК). Если отсутствуют → 🔴 (договор может быть признан незаключённым).

#### 2.3. Check судебную практику

Если `kad-mcp` доступен — для критичных пунктов (LoL, force majeure, неустойка, ответственность, IP) делай `kad.search_cases(query="<ключевые слова из пункта>", limit=5)`.

Особое внимание:
- Постановления Пленума ВС РФ — высший приоритет (если найдены — цитируй обязательно).
- Решения арбитражных судов округа того, где будет рассмотрение спора по подсудности (если задана).

#### 2.4. Risk assessment

Каждому пункту присвой светофор:

- 🔴 **КРИТИЧНО** — нарушение императивной нормы, отсутствие существенного условия, явно неравноправная формулировка с большим финансовым риском, нарушение публичных норм (152-ФЗ, 115-ФЗ, санкционные).
- 🟡 **ВНИМАНИЕ** — формулировка содержит риск, но при определённых обстоятельствах может работать. Допустима после согласования с lawyer-in-the-loop и/или внесения уточнений.
- 🟢 **OK** — стандартная формулировка, соответствует ГК и практике, без значимого риска.

Если профиль указывает sell-side / buy-side — добавь **сторону-чьи-интересы-затронуты**:
- 🔴(us) — риск нашей стороне (заполняется из profile)
- 🔴(them) — риск контрагенту, что может быть тактическим преимуществом для нас
- 🔴(both) — обоюдный риск

#### 2.5. Suggest redline

Для каждого 🔴 и 🟡 пункта **обязательно** предложи альтернативную формулировку:

```
Recommended redline:
> [предложенный текст пункта]

Обоснование: со ссылкой на конкретную статью ГК РФ + (если есть) судебную практику.
```

Redline должен быть:
- Готов к копированию (без квадратных скобок-заглушек, кроме переменных типа `[сумма]`)
- Юридически корректным
- Сохраняющим business intent оригинала, если он не противоправен

### Шаг 3. Cross-clause analysis

После пунктового анализа — обязательно проверь взаимодействия:

- **Противоречия:** пункт X.1 говорит одно, X.2 — другое (например, разные сроки оплаты в разных разделах).
- **Когерентность ответственности:** LoL в пункте про общую ответственность должен соответствовать пунктам про неустойку, indemnity, штрафы. Они могут де-факто обходить LoL.
- **Сроки:** все сроки (срок поставки, оплаты, расторжения, претензионный) должны быть согласованы.
- **Терминология:** одна и та же сущность называется одинаково по всему документу. Если "Заказчик" в одном месте, "Клиент" в другом — флаг 🟡.
- **Применимое право vs подсудность:** "право РФ + подсудность Лондона" — потенциально 🔴 (несовместимая конструкция в большинстве случаев).

### Шаг 4. Compliance checks (cross-cutting)

#### 4.1. 152-ФЗ (Персональные данные)

Если договор включает обработку ПДн (любая работа с информацией о физлицах):
- Есть ли согласие субъекта ПДн?
- Назначен ли оператор ПДн?
- Соответствие cross-border передачи правилам 152-ФЗ (запрет передачи в "плохие" страны без письменного согласия)?
- Соблюдение localisation (хранение в РФ для российских граждан)?

#### 4.2. 115-ФЗ (ПОД-ФТ)

Если контрагент — иностранное лицо или сумма выше KYC-порога из profile:
- Проведена ли KYC?
- Есть ли в договоре положение об обязательстве предоставить документы для compliance?
- Sanctions screening (US OFAC / EU / UK / RU контр-санкционный список)?

#### 4.3. НК РФ (Налоги)

- НДС правильно учтён в цене (с учётом / без учёта НДС — должно быть явно указано)?
- Если контрагент применяет УСН — кто платит НДС?
- Если cross-border — правильное применение СИДН?

#### 4.4. Санкции

Если контрагент — иностранное лицо ИЛИ Российская компания с иностранным контролем:
- Запросить через `egrul-mcp` бенефициаров, проверить по sanctions lists (этот шаг пока manual — TODO для `sanctions-mcp`).

## Что этот skill НЕ делает

- ❌ **Не подписывает договор** и не отправляет его. Только анализ.
- ❌ **Не работает с скан-PDF без OCR.** Сначала текст должен быть экстрактирован.
- ❌ **Не даёт окончательное legal advice.** Все выводы — preliminary, требуют верификации юристом.
- ❌ **Не оценивает personalities.** Если контрагент засветился в негативном контексте в КАД — только факты дел, без характеристики.
- ❌ **Не пишет иски / претензии.** Для этого нужен `litigation` pack.

---

## Critical guardrails

- **Дата НПА.** Все цитаты на ГК РФ и др. НПА — ОБЯЗАТЕЛЬНО с указанием редакции (через `pravo-mcp`) или флага `[ред. требует проверки]`. НПА меняются, цитата без даты — ловушка.
- **Юрисдикция.** Если в договоре `governing law` — не РФ, прямо предупреди и предложи переключиться на соответствующий специализированный skill (которого у нас пока нет; в этом случае — рекомендуй внешнего адвоката).
- **Конфиденциальность.** Текст договора и output не отправлять никуда, кроме локального файла. Никаких external API кроме тех MCP, что подключены пользователем явно.
- **Sanctions.** Если контрагент — иностранное лицо: предупреди, что пользователь обязан самостоятельно проверить sanctions lists. Skill этого не делает в MVP.
- **No legal advice claim.** В резюме и в каждом отчёте — disclaimer про preliminary анализ.

---

## Важно

- **Уровень детализации зависит от роли пользователя из profile.** Для юриста — полный технический отчёт. Для non-lawyer — короткое резюме + топ-3 рисков, без юридического жаргона.
- **Если MCP servers недоступны** — продолжай в degraded mode, но честно укажи в финальном отчёте, что:
  - "Редакции НПА не проверены (pravo-mcp недоступен) — рекомендуется верификация юристом."
  - "Контрагент не проверен в ЕГРЮЛ (egrul-mcp недоступен)."
  - "Судебная практика не проверена (kad-mcp недоступен)."
- **Учитывай длину.** Очень длинные договоры (50+ страниц) — предложи разбить на разделы и анализировать по частям, либо запусти full review с предупреждением о времени.

## ⚠ Юридический disclaimer

Данный skill — **техническая платформа**, не оказывает юридических услуг по ст.2
ФЗ-63 «Об адвокатской деятельности». Outputs **не заменяют** консультацию
лицензированного юриста / адвоката ФПА.

AI может галлюцинировать, выдавать устаревшие нормы, неверно интерпретировать
факты. Material decisions (увольнение, M&A, налоговый спор, IP litigation) —
**обязательно** engage outside-адв ФПА с релевантной специализацией.

Проект `ru-legal` и его contributors не несут ответственности за решения,
принятые на основе outputs системы. Использование — на свой риск.
