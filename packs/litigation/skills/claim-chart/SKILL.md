# /claim-chart (civil mode — доказательственная матрица)

## Назначение

В РФ litigation — **для каждого элемента состава требования** нужны доказательства.
Этот skill строит таблицу: requirement element → факт → доказательство → adequacy.

**Patent mode** (US patent claim mapping) **skipped** — РФ патентное процесс
существенно отличается (специальные нормы ГК ч.4, Суд по интеллектуальным правам).

## Workflow

### Шаг 1. Identify claim type

| Тип иска | Состав требования (элементы) |
|----------|-------------------------------|
| **Взыскание долга (договор)** | 1) Существование обязательства; 2) Срок исполнения наступил; 3) Должник не исполнил; 4) Размер долга |
| **Расторжение договора (ст.450)** | 1) Существование договора; 2) Существенное нарушение или иное основание; 3) Соблюдён претензионный порядок |
| **Возмещение убытков (ст.15+393)** | 1) Противоправность действий; 2) Размер убытков; 3) Причинная связь; 4) Вина (для intentional) |
| **Неосновательное обогащение (ст.1102)** | 1) Приобретение / сбережение имущества; 2) Без правового основания; 3) Размер; 4) За счёт истца |
| **Признание сделки недействительной** | По основанию: ст.166-179 ГК (различные дефекты) |
| **Принуждение к исполнению в натуре** | 1) Существование обязательства; 2) Возможность исполнения в натуре; 3) Истец не получил исполнение |
| **Защита деловой репутации (ст.152)** | 1) Распространение сведений; 2) Не соответствуют действительности; 3) Порочат репутацию |

Для **антимонопольных, налоговых, корпоративных** — другие специфические составы.

### Шаг 2. Map elements → facts

Для каждого элемента — какие конкретные **факты** мы должны доказать:

```yaml
claim_type: contract-debt-collection
elements:
  - id: existence_of_obligation
    description: "Существование обязательства"
    facts_required:
      - "Заключён договор поставки № 42 от 15.03.2024"
      - "Стороны: ООО Истец и ООО Ответчик (правомочны)"
      - "Договор не расторгнут / не признан недействительным"

  - id: deadline_passed
    description: "Срок исполнения наступил"
    facts_required:
      - "Согласно п. 3.2 Договора — оплата в течение 30 банковских дней после акта"
      - "Акт приёма-передачи подписан 01.06.2024"
      - "31.07.2024 — крайний срок оплаты"
      - "Дата на момент иска — после 31.07.2024"

  - id: non_performance
    description: "Должник не исполнил"
    facts_required:
      - "Платёж не поступил на расчётный счёт Истца"
      - "Backend bank statement подтверждает отсутствие платежа"
      - "Должник не представлял документов об альтернативном исполнении"

  - id: amount
    description: "Размер долга"
    facts_required:
      - "Сумма к оплате — 5 250 000 руб (из акта приёма-передачи)"
      - "Расчёт процентов по ст. 395 ГК — приложен"
      - "Договорная неустойка — расчёт за период просрочки"
```

### Шаг 3. Map facts → evidence

Для каждого факта — какие **документы / свидетели** его подтверждают:

```yaml
- fact: "Заключён договор поставки № 42 от 15.03.2024"
  evidence:
    - type: document
      description: "Договор № 42 (оригинал)"
      source: "contracts/PD-42-2024.pdf"
      reliability: high  # подписан обеими сторонами + печати + сканы с EDS
    - type: witness
      description: "Иванов И.И. — представитель Истца на подписании"
      reliability: medium  # дополнительное подтверждение

- fact: "Платёж не поступил на расчётный счёт Истца"
  evidence:
    - type: document
      description: "Выписка с расчётного счёта за период 01.06.2024 — 18.05.2026"
      source: "bank-statement-2024-2026.pdf"
      reliability: high  # официальный banking документ
    - type: document
      description: "Справка из банка об отсутствии поступлений от ИНН ответчика"
      source: "bank-cert-2026-05-15.pdf"
      reliability: high
```

### Шаг 4. Adequacy check

Для каждого элемента — оценка достаточности доказательств:

| Элемент | Доказательств | Adequacy |
|---------|--------------|----------|
| Существование обязательства | Договор + копия с подписями | 🟢 Strong |
| Срок наступил | Договор п.3.2 + акт приёма-передачи | 🟢 Strong |
| Должник не исполнил | Выписка банка + справка | 🟢 Strong |
| Размер долга | Расчёт + первичка | 🟢 Strong |

🟢 = достаточно для суда
🟡 = желательно усилить (witness, дополнительный doc)
🔴 = gap critical — рискуем проиграть на этом элементе

### Шаг 5. Priority output: **gap list**

Главное в claim-chart — это **gaps**. Что нам нужно дополнительно?

```markdown
## Gaps (необходимо собрать дополнительно)

### 🔴 Critical (без этого слабо)
- [ ] Гарантийное письмо от должника об осведомлённости о долге (если есть)
- [ ] Переписка по teh email о просьбах оплаты — может укрепить факт уведомления

### 🟡 Strengthening (улучшит позицию)
- [ ] Witness statement от Иванова И.И.
- [ ] Дополнительные shipping documents (если были партии товара)
```

### Шаг 6. Output to file

```markdown
matters/[slug]/claim-chart-[date].md
```

С полной матрицей elements → facts → evidence + gap list.

## Что НЕ делает

- **Patent mode skipped** — РФ патенты через Суд по интеллектуальным правам с
  спец.процедурой; не покрыто
- Не collects сами documents — только identifies необходимые
- Не оценивает strength доказательства strictly юридически — preliminary
- Не reлates к конкретным судебным решениям без instruct'ивности
